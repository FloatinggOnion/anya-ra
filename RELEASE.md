# Anya Release & Packaging Guide

## Overview

Your Anya project is now configured for automated, cross-platform releases using GitHub Actions. This guide explains how to package and release Anya for macOS, Windows, and Linux.

---

## Supported Platforms & Architectures

### macOS
- **Apple Silicon (M1/M2/M3)** - aarch64
- **Intel Mac** - x86_64
- Format: `.dmg` (Disk Image installer)

### Windows
- **64-bit (x64)** - x86_64
- **32-bit (x86)** - i686  
- **ARM64** - aarch64
- Format: `.msi` (Windows Installer)

### Linux
- **64-bit (x86_64)** - x86_64
- **ARM64** - aarch64
- Format: `.AppImage` (AppImage executable)

---

## Automated Release Workflow

### Prerequisites
1. Push code to GitHub repository
2. Ensure all tests pass (they run automatically)
3. Create a version tag

### Step 1: Tag a Release

```bash
# Create and push a version tag
git tag -a v0.2.0 -m "Release version 0.2.0: Add feature X and fix Y"
git push origin v0.2.0
```

**Important**: Tag format must start with `v` (e.g., `v0.2.0`, `v1.0.0`)

### Step 2: GitHub Actions Builds

Once the tag is pushed:

1. **7 builds start in parallel** (one for each platform/arch combo)
2. **Each job**:
   - Checks out your code
   - Installs dependencies
   - Compiles frontend (TypeScript + Svelte)
   - Compiles backend (Rust)
   - Creates platform-specific installer
   - Uploads artifact

3. **Release job** (after all builds complete):
   - Collects all platform binaries
   - Creates a GitHub Release page
   - Uploads files with user-friendly names
   - Generates release notes

### Step 3: Download & Share

Users visit: `https://github.com/yourusername/anya-ra/releases`

They see all binaries organized by platform:
- macOS users download `.dmg`
- Windows users download `.msi`
- Linux users download `.AppImage`

## Manual Local Build (for Testing)

If you want to build locally before releasing:

```bash
# Build for current platform only
pnpm tauri build

# Output location:
# macOS: src-tauri/target/release/bundle/dmg/
# Windows: src-tauri/target/release/bundle/msi/
# Linux: src-tauri/target/release/bundle/appimage/
```

## Release Checklist

Before tagging a release:

- [ ] Update version in `package.json` and `src-tauri/tauri.conf.json`
- [ ] Update `CHANGELOG.md` (if using one)
- [ ] Run tests: `pnpm test`
- [ ] Build locally: `pnpm tauri build`
- [ ] Test the built application
- [ ] Commit changes: `git commit -am "chore: Bump version to v0.2.0"`
- [ ] Create tag: `git tag -a v0.2.0 -m "Release message"`
- [ ] Push tag: `git push origin v0.2.0`
- [ ] Check GitHub Actions: `https://github.com/yourusername/anya-ra/actions`
- [ ] Verify release on GitHub Releases page
- [ ] Share release announcement

## Troubleshooting

### Build Fails on GitHub Actions

Check the workflow logs:
1. Go to Actions tab in your GitHub repo
2. Click the failed workflow run
3. Click the failed job to see detailed logs
4. Common issues:
   - Missing system dependencies (fixed in workflow)
   - Rust toolchain issues (use dtolnay/rust-toolchain)
   - Node.js version mismatch (use Node 20)

### Release Not Created

Ensure:
- Tag starts with `v` (e.g., `v0.2.0`)
- All build jobs completed successfully
- No permission issues with GITHUB_TOKEN (automatic in workflows)

### Size Issues

If binaries are too large:
- Check `src-tauri/tauri.conf.json` has LTO enabled
- Ensure release profile has `opt-level = "z"`
- Consider code splitting in Vite for frontend

## Version Bumping Strategy

Recommend semantic versioning:

- **MAJOR** (v1.0.0): Breaking changes
- **MINOR** (v0.2.0): New features, backward compatible
- **PATCH** (v0.1.1): Bug fixes only

Update in two places:

```bash
# 1. package.json
"version": "0.2.0"

# 2. src-tauri/tauri.conf.json
"version": "0.2.0"
```
